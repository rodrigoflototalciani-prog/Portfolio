The problem I solved

After the chatbot went live, manually reviewing hundreds of daily sessions was not scalable. Problems went undetected for days. There was no visibility into which topics caused friction or whether the bot was giving factually correct information. I built this agent independently — no engineering support — to close that gap.

How it works
Reads every new session from Google Sheets, deduplicates against data warehouse
Step 1: GPT-4o-mini scores quality (0–100), detects sentiment, resolution, bot failures, language requests

Writes structured results to data warehouse, feeds Looker Studio dashboard

Key code — quality scoring prompt (Google Apps Script)

The most critical part of the agent is the LLM prompt that scores each session. Hard ceiling rules are baked in so scores stay consistent regardless of which model version runs. The prompt instructs the LLM to return structured JSON only — no prose — which is then parsed and written directly to BigQuery.

// Step 1: quality analysis — sent to GPT-4o-mini for every session const prompt = `You are a strict quality analyst evaluating chatbot conversations. Your job is to find problems, not to be nice. Analyze this conversation and return ONLY valid JSON — no markdown, no explanation. STEP 1 — Read the full conversation carefully. STEP 2 — Look for these problems: - Did the user have to ask the same thing more than once? - Did the bot give an incomplete or vague answer? - Did the bot fail to understand the actual intent? - Did the user leave without a clear answer? - Did the user ask about changing app language? (any language) → language_request = true STEP 3 — Score honestly. Most conversations have at least one problem. Return exactly this JSON: { "topic": "activation|usage|inclusions_exclusions|pricing|refunds|technical_issue|other", "sentiment": "positive|neutral|negative", "resolution": "resolved|partial|unresolved|escalated", "bot_failure": true or false, "bot_failure_reason": "specific explanation or null", "quality_score": integer 0-100, "quality_reason": "specific problems found, max 15 words", "language_request": true or false } // Hard scoring rules — prevents LLM from being too generous - negative sentiment → maximum score 45 - neutral sentiment → maximum score 65 - unresolved conversation → maximum score 40 - user had to repeat themselves → maximum score 45 - bot asked for info it already had → maximum score 50`; // Call GPT-4o-mini, parse JSON response, combine with accuracy check const response = UrlFetchApp.fetch('https://api.openai.com/v1/chat/completions', { method: 'post', headers: { 'Authorization': 'Bearer ' + OPENAI_KEY }, payload: JSON.stringify({ model: 'gpt-4o-mini', temperature: 0.1, // low temp = consistent scoring messages: [{ role: 'user', content: prompt }] }) }); const parsed = JSON.parse(cleanResponse(response)); // Step 2: accuracy check — second LLM pass for venue facts const accuracy = checkLocationAccuracy(sessionId, conversation); // Combine: if accuracy fails, apply penalty to quality score const finalScore = Math.max(0, (parsed.quality_score || 50) - (accuracy.accuracy_score_penalty || 0) );

Multi-Agent System — Data Retrieval Accuracy

A production multi-agent pipeline was returning "information not available" on 40% of valid queries. Diagnosed the failure mode and cut false negatives by 50% through prompt engineering alone — no code changes.

Multi-agentLLM orchestrationMCP toolsPrompt engineering
Result

false negatives
The system

A production multi-agent AI system uses four specialised agents: a chat agent that orchestrates the conversation, a data collection agent that retrieves records via MCP tools, a data processing agent that shapes the data into a response, and a data querying agent that executes against an in-memory database.

The problem

The system responded with "information not available" on ~40% of queries where the data actually existed. Users were told the system couldn't help them when it could. The root cause was not missing data — it was the agents' retrieval behaviour.

Root cause identified
Agents were concluding "not available" before exhausting the full retrieval chain — a tool-calling behaviour problem, not a data problem
My approach
Diagnosis: Reviewed failure patterns to identify which agent in the chain was breaking down
Hypotheses: Formed testable theories about prompt instructions causing premature conclusions
Iteration: Added explicit instructions — agents must exhaust all available tools before concluding unavailable
Reasoning: Added step-by-step guidance to ensure agents complete the full retrieval chain
Testing: Validated each iteration against known failure cases

Personalised Recommendation Engine PoC

Built a working recommendation engine to make the business case to management — solving cold start for one-time tourists and popularity bias across 100+ venues using collaborative filtering.

Pythonscikit-learnCollaborative filteringCold startIDF weighting
Purpose
Business case for MT investment
The problem

The chatbot was recommending the same 10–15 famous venues to every tourist, ignoring 100+ others. Two business problems: tourists got a generic experience, and most venue partners received almost zero referral traffic from the app. The core technical challenge: most tourists are one-time visitors — there is no visit history to learn from on day one.

Three-layer solution
Cold start: Onboarding traveller type (Explorer / Art Lover / History Buff / Science Enthusiast) gives instant personalisation before any visits
Collaborative filtering: As visits accumulate, cosine similarity finds past tourists with similar patterns — weighted by similarity score
IDF weighting: log(N/visits) boosts rare venues so they compete fairly with popular ones
Category cap: Max 2 per category forces variety — no five art museums in a row
Core scoring logic
# Similarity-weighted CF score score = Σ(similarity × rating) / Σ(similarity) # IDF — reduces popularity bias idf = log(N / visit_count) final = (1-w) × score + w × score × idf_norm # Cold start confidence blend cf_conf = min(1.0, len(visited) / 5) score = cf_conf × cf_score + (1 - cf_conf) × popularity + profile_boost

Key code — cold start profile boost (Python)

This function converts the onboarding traveller type directly into a recommendation score boost. Venues shared across multiple profiles accumulate stacked boosts automatically — no special logic needed. This is what makes personalisation possible from minute one, before any visit history exists.

# Traveller types from onboarding — mapped to real venue lists TRAVELLER_PROFILES = { "Explorer": ["City Canal Cruise", "ARTIS Zoo", "Zaanse Schans", ...], "Art Lover": ["Foam Photography Museum", "Stedelijk Museum", "STRAAT", ...], "History Buff": ["Rijksmuseum", "Rembrandt House", "National Maritime Museum", ...], "Science Enthusiast": ["NEMO Science Museum", "ARTIS-Micropia", "Teylers Museum", ...], } def build_profile_boost(*traveller_types: str) -> dict: """ Convert onboarding traveller type(s) into a score boost dict. Venues in multiple profiles get stacked boosts automatically. Example: build_profile_boost("Art Lover") → {"Foam Photography Museum": 0.3, "Stedelijk Museum": 0.3, ...} build_profile_boost("Art Lover", "Science Enthusiast") → {"Nxt Museum": 0.6, ...} ← appears in both, boost doubles """ boost = {} for t in traveller_types: matched = next( (k for k in TRAVELLER_PROFILES if k.lower() == t.lower()), None ) if not matched: print(f"[WARN] Unknown type '{t}'") continue for venue in TRAVELLER_PROFILES[matched]: boost[venue] = boost.get(venue, 0) + 0.3 # stack if in multiple profiles return boost # Usage — called at cold start when user has zero visit history boost = build_profile_boost("Art Lover") recs = recommender.recommend_cold(visited=[], profile_boost=boost, n=5) # As visits accumulate, CF confidence grows and blends in cf_confidence = min(1.0, len(visited) / 5) score = cf_confidence * cf_score \ + (1 - cf_confidence) * popularity \ + profile_boost.get(venue, 0)
